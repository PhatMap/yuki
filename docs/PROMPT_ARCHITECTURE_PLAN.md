# Prompt Architecture Plan

Prompt handling is core infrastructure for Yuki.

Every AI task must have a clear, versioned, editable, testable prompt contract. Prompts must not be hidden as one-off strings inside random components or providers.

## Why this matters

Yuki depends on AI for:

- Chapter Scout;
- arc mapping;
- deep story analysis;
- Story Bible extraction;
- Timeline extraction;
- Relationship extraction;
- World Tracker extraction;
- rewrite impact planning;
- rewrite draft generation;
- future Reader Persona;
- future Role-play.

If prompts are loose, Yuki will become unstable and hard to improve.

## Product rule

All prompts sent to AI should be managed through Prompt Manager or a prompt registry layer.

Each AI task should have:

1. Prompt ID.
2. Version.
3. Category.
4. Purpose.
5. System instruction.
6. Editable user instruction.
7. Locked output contract.
8. Variable schema.
9. Input builder.
10. Output schema validator.
11. Runtime metadata.
12. Test sample.
13. Reset-to-default support.

## Prompt layers

### 1. System identity

Global Yuki identity.

Purpose:

- define Yuki as a long-story analysis and continuity OS;
- require canon discipline;
- require uncertainty handling;
- ban unsupported invention.

Example prompt ID:

- `story-system-identity`

### 2. Task prompt

Task-specific instruction.

Examples:

- `chapter-scout`
- `arc-map`
- `import-analysis`
- `story-bible-extract`
- `timeline-extract`
- `relationship-extract`
- `world-tracker-extract`
- `rewrite-impact-planner`
- `rewrite-draft`
- `reader-persona-question-plan`
- `reader-persona-profile-generate`
- `role-play-strict-canon-chat`

### 3. Input payload

Structured JSON payload generated by code.

Prompt text should not rely on vague natural language alone. AI should receive explicit data blocks.

Examples:

- chapter samples;
- full selected chapters;
- existing analysis result;
- selected branch change;
- canon constraints;
- Reader Persona profile;
- role-play session mode.

### 4. Locked contract

Non-editable section.

Purpose:

- output JSON schema;
- strict response format;
- required fields;
- forbidden behavior;
- safety rules;
- no markdown if JSON is expected.

The user can edit task wording, but should not accidentally break the required output contract.

## Editable vs locked sections

### Editable by user

- task tone;
- extraction emphasis;
- domain-specific rules;
- what to prioritize;
- optional wording.

### Locked by system

- output schema;
- required keys;
- JSON-only requirement;
- validation-critical instructions;
- no unsupported facts;
- chapter coverage rules;
- safety/content handling policy.

## Prompt Manager requirements

Prompt Manager should support:

- list prompts by category;
- show prompt ID and version;
- edit prompt body;
- preview variables;
- preview final rendered prompt;
- reset one prompt;
- reset all prompts;
- mark locked contract separately;
- show which app feature uses each prompt;
- show last updated time;
- export/import prompt templates later.

## Required core prompts for Priority 1

### 1. `chapter-scout`

Purpose:

- read chapter samples only;
- classify chapters before deep analysis;
- recommend skip/light/deep load.

Input:

```ts
{
  goal: "story_bible" | "rewrite" | "canon";
  adultSignalPolicy: string;
  chapterSamples: ChapterSample[];
}
```

Output:

```ts
{
  results: ChapterScoutResult[];
}
```

Rules:

- return exactly one result for each chapter sample;
- no missing chapterIndex;
- no markdown;
- reason in Vietnamese;
- do not use keyword-only matching;
- infer from event, relationship, state, reveal, worldbuilding, ending hook.

### 2. `arc-map`

Purpose:

- group scout results into story arcs;
- recommend which arcs need deep analysis;
- use Scout metadata only, not invented full chapter content.

Input:

```ts
{
  windowLabel: string;
  chapterCount: number;
  scoutResults: ChapterScoutResult[];
}
```

Output:

```ts
{
  arcs: ArcMapItem[];
}
```

Rules:

- no invented chapter content;
- title/summary/whyLoad in Vietnamese;
- recommendedDeepChapters must come from provided scoutResults.

### 3. `import-analysis` / `deep-story-analysis`

Purpose:

- analyze selected full chapter/chunk content;
- extract Canon-relevant data.

Input:

- story metadata;
- selected chapters;
- selected chunks;
- optional scout/arc context.

Output:

- `StoryAnalysisResult`.

Rules:

- include chapter references;
- mark uncertainty;
- use only provided text;
- do not invent missing data;
- return strict JSON.

### 4. `story-bible-extract`

Purpose:

- transform analysis data into durable Story Bible facts.

Output should separate:

- confirmed fact;
- inferred fact;
- uncertain fact;
- contradiction.

### 5. `rewrite-impact-planner`

Purpose:

- plan downstream effects of a change request;
- do not rewrite prose.

### 6. `rewrite-draft`

Purpose:

- generate revised chapter prose using approved plan and canon constraints.

## Prompt runtime requirements

Prompt runtime should produce:

```ts
interface RenderedPromptPackage {
  promptId: string;
  promptVersion: number;
  category: string;
  systemInstruction: string;
  editableInstruction: string;
  lockedContract: string;
  variables: Record<string, unknown>;
  renderedPrompt: string;
  inputPayload: unknown;
  outputSchemaId: string;
  missingVariables: string[];
  estimatedTokens?: number;
}
```

AI calls should log/store prompt metadata:

- prompt ID;
- prompt version;
- provider;
- model;
- endpoint;
- input size;
- output validation result;
- createdAt/completedAt;
- cache key.

Do not store API keys in prompt metadata.

## Output validation rule

Every JSON AI task must have a validator.

Required flow:

```txt
AI response → extract JSON → validate schema → accept/save or reject with error
```

Never save invalid AI output as canon data.

## Current known gap

As of this plan, Yuki already has a basic Prompt Manager and prompt registry, but it is not strict enough for long-term scale.

Known problems:

- prompt templates are too coarse;
- `import-analysis` needs stronger chapter/chunk context handling;
- Chapter Scout and Arc Map prompts are missing;
- output schema contracts need to be stricter;
- prompt preview/test tooling is not complete;
- prompt metadata should be easier to inspect during AI jobs.

## Implementation phase order

### Phase 1 — Fix current analysis prompt path

- Ensure Gemini final prompt includes actual story/chapter/chunk context.
- Improve `import-analysis` prompt.
- Add validation guard against contextless prompt.

### Phase 2 — Add Chapter Scout and Arc Map prompts

- Add prompt templates.
- Add input builders.
- Add output validators.
- Keep UI simple.

### Phase 3 — Prompt Manager upgrade

- Show prompt ID/version/category.
- Show where each prompt is used.
- Add final rendered prompt preview.
- Add reset-to-default clearly.

### Phase 4 — Prompt test harness

- Allow testing a prompt with sample payload.
- Show parsed output and validation result.
- No save to canon unless user runs real workflow.

### Phase 5 — Expand to Persona and Role-play

- Add prompt groups only after Priority 1 is stable.
