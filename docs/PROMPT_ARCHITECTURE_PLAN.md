# Prompt Architecture Plan

Prompt handling is core infrastructure for Yuki.

Every AI task must have a clear, versioned, editable, testable prompt contract. Prompts must not be hidden as one-off strings inside random components or providers.

## Why this matters

Yuki depends on AI for:

- Chapter Scout;
- arc mapping;
- deep story analysis;
- Story Bible extraction;
- Timeline extraction;
- Relationship extraction;
- World Tracker extraction;
- rewrite impact planning;
- rewrite draft generation;
- future Reader Persona;
- future Role-play.

If prompts are loose, Yuki will become unstable and hard to improve.

## Product rule

All prompts sent to AI should be managed through Prompt Manager or a prompt registry layer.

Each AI task should have:

1. Prompt ID.
2. Version.
3. Category.
4. Scope: global or story-specific.
5. Purpose.
6. System instruction.
7. Editable user instruction.
8. Locked output contract.
9. Variable schema.
10. Input builder.
11. Output schema validator.
12. Runtime metadata.
13. Test sample.
14. Reset-to-default support.

## Prompt scope model

Yuki needs two prompt scopes.

### 1. Global prompts

Global prompts apply across the whole app.

Use global prompts for:

- Yuki system identity;
- default task instructions;
- default JSON output contracts;
- shared analysis rules;
- shared rewrite rules;
- default Scout / Arc Map / Deep Analysis prompts;
- default Persona / Role-play prompts later.

Storage direction:

- global prompt templates live in the existing global prompt template store;
- they are managed by Prompt Manager;
- resetting a global prompt restores the shipped default version.

Examples:

- `story-system-identity`
- `chapter-scout`
- `arc-map`
- `deep-story-analysis`
- `rewrite-impact-planner`
- `rewrite-draft`

### 2. Story-specific prompts

Story-specific prompts override or extend global prompts for one imported story.

Use story-specific prompts for:

- story-specific canon rules;
- genre-specific extraction rules;
- naming conventions;
- terms that must be preserved;
- content the user wants Yuki to pay special attention to;
- custom rewrite constraints for that story;
- story-specific no-go rules;
- special relationship/worldbuilding rules.

Story-specific prompts must not replace locked output contracts. They should be inserted as an editable `story instruction` layer before the input payload.

Storage direction:

- story-specific prompt overrides should be stored in IndexedDB;
- key by `storyId + promptId`;
- fall back to global prompt when no story override exists;
- allow reset story override to global default;
- do not store story prompt overrides in localStorage.

Examples:

- `story:{storyId}:chapter-scout`
- `story:{storyId}:deep-story-analysis`
- `story:{storyId}:rewrite-impact-planner`
- `story:{storyId}:rewrite-draft`

## Prompt composition order

Final prompt package should be composed in this order:

```txt
1. Global system identity
2. Global task instruction
3. Story-specific instruction override/extension
4. Locked output contract
5. Structured input payload
```

Rules:

- locked contract always wins;
- story-specific prompt cannot remove required output fields;
- global defaults must work even when story-specific prompt is empty;
- final rendered prompt preview must show all layers;
- runtime metadata must record which layers were used.

## Prompt layers

### 1. System identity

Global Yuki identity.

Purpose:

- define Yuki as a long-story analysis and continuity OS;
- require canon discipline;
- require uncertainty handling;
- ban unsupported invention.

Example prompt ID:

- `story-system-identity`

### 2. Task prompt

Task-specific instruction.

Examples:

- `chapter-scout`
- `arc-map`
- `import-analysis`
- `story-bible-extract`
- `timeline-extract`
- `relationship-extract`
- `world-tracker-extract`
- `rewrite-impact-planner`
- `rewrite-draft`
- `reader-persona-question-plan`
- `reader-persona-profile-generate`
- `role-play-strict-canon-chat`

### 3. Story instruction

Story-specific editable instruction.

Purpose:

- tell AI what matters for this story;
- preserve special terms and names;
- define custom canon sensitivity;
- adjust extraction priorities for this story;
- record user preferences that only apply to this story.

Examples:

- `Ưu tiên theo dõi quan hệ thầy trò và các lần đổi phe.`
- `Không Việt hóa tên chiêu thức Hán Việt.`
- `Khi phân tích, đánh dấu rõ các lần nhân vật chính dùng danh tính giả.`

### 4. Input payload

Structured JSON payload generated by code.

Prompt text should not rely on vague natural language alone. AI should receive explicit data blocks.

Examples:

- chapter samples;
- full selected chapters;
- existing analysis result;
- selected branch change;
- canon constraints;
- Reader Persona profile;
- role-play session mode.

### 5. Locked contract

Non-editable section.

Purpose:

- output JSON schema;
- strict response format;
- required fields;
- forbidden behavior;
- safety rules;
- no markdown if JSON is expected.

The user can edit task wording and story instructions, but should not accidentally break the required output contract.

## Editable vs locked sections

### Editable by user

- global task wording;
- story-specific instruction;
- task tone;
- extraction emphasis;
- domain-specific rules;
- what to prioritize;
- optional wording.

### Locked by system

- output schema;
- required keys;
- JSON-only requirement;
- validation-critical instructions;
- no unsupported facts;
- chapter coverage rules;
- safety/content handling policy.

## Prompt Manager requirements

Prompt Manager should support:

- list prompts by category;
- show prompt ID, version, and scope;
- switch between global prompts and story-specific overrides;
- edit global prompt body;
- edit story-specific instruction for a selected story;
- preview variables;
- preview final rendered prompt;
- reset one global prompt;
- reset one story override;
- reset all global prompts;
- mark locked contract separately;
- show which app feature uses each prompt;
- show last updated time;
- export/import prompt templates later.

## Required core prompts for Priority 1

### 1. `chapter-scout`

Purpose:

- read chapter samples only;
- classify chapters before deep analysis;
- recommend skip/light/deep load.

Input:

```ts
{
  goal: "story_bible" | "rewrite" | "canon";
  adultSignalPolicy: string;
  storyInstruction?: string;
  chapterSamples: ChapterSample[];
}
```

Output:

```ts
{
  results: ChapterScoutResult[];
}
```

Rules:

- return exactly one result for each chapter sample;
- no missing chapterIndex;
- no markdown;
- reason in Vietnamese;
- do not use keyword-only matching;
- infer from event, relationship, state, reveal, worldbuilding, ending hook.

### 2. `arc-map`

Purpose:

- group scout results into story arcs;
- recommend which arcs need deep analysis;
- use Scout metadata only, not invented full chapter content.

Input:

```ts
{
  windowLabel: string;
  chapterCount: number;
  storyInstruction?: string;
  scoutResults: ChapterScoutResult[];
}
```

Output:

```ts
{
  arcs: ArcMapItem[];
}
```

Rules:

- no invented chapter content;
- title/summary/whyLoad in Vietnamese;
- recommendedDeepChapters must come from provided scoutResults.

### 3. `import-analysis` / `deep-story-analysis`

Purpose:

- analyze selected full chapter/chunk content;
- extract Canon-relevant data.

Input:

- story metadata;
- story-specific instruction;
- selected chapters;
- selected chunks;
- optional scout/arc context.

Output:

- `StoryAnalysisResult`.

Rules:

- include chapter references;
- mark uncertainty;
- use only provided text;
- do not invent missing data;
- return strict JSON.

### 4. `story-bible-extract`

Purpose:

- transform analysis data into durable Story Bible facts.

Output should separate:

- confirmed fact;
- inferred fact;
- uncertain fact;
- contradiction.

### 5. `rewrite-impact-planner`

Purpose:

- plan downstream effects of a change request;
- do not rewrite prose.

### 6. `rewrite-draft`

Purpose:

- generate revised chapter prose using approved plan and canon constraints.

## Prompt runtime requirements

Prompt runtime should produce:

```ts
interface RenderedPromptPackage {
  promptId: string;
  promptVersion: number;
  category: string;
  scope: "global" | "story-specific";
  storyId?: string;
  systemInstruction: string;
  globalTaskInstruction: string;
  storyInstruction?: string;
  lockedContract: string;
  variables: Record<string, unknown>;
  renderedPrompt: string;
  inputPayload: unknown;
  outputSchemaId: string;
  missingVariables: string[];
  estimatedTokens?: number;
}
```

AI calls should log/store prompt metadata:

- prompt ID;
- prompt version;
- prompt scope;
- story override id if used;
- provider;
- model;
- endpoint;
- input size;
- output validation result;
- createdAt/completedAt;
- cache key.

Do not store API keys in prompt metadata.

## Output validation rule

Every JSON AI task must have a validator.

Required flow:

```txt
AI response → extract JSON → validate schema → accept/save or reject with error
```

Never save invalid AI output as canon data.

## Current known gap

As of this plan, Yuki already has a basic Prompt Manager and prompt registry, but it is not strict enough for long-term scale.

Known problems:

- prompt templates are too coarse;
- global and story-specific prompt layers are not fully separated;
- story-specific prompt overrides need IndexedDB support;
- `import-analysis` needs stronger chapter/chunk context handling;
- Chapter Scout and Arc Map prompts are missing;
- output schema contracts need to be stricter;
- prompt preview/test tooling is not complete;
- prompt metadata should be easier to inspect during AI jobs.

## Implementation phase order

### Phase 1 — Fix current analysis prompt path

- Ensure Gemini final prompt includes actual story/chapter/chunk context.
- Improve `import-analysis` prompt.
- Add validation guard against contextless prompt.

### Phase 2 — Add Chapter Scout and Arc Map prompts

- Add prompt templates.
- Add input builders.
- Add output validators.
- Keep UI simple.

### Phase 3 — Add story-specific prompt overrides

- Add IndexedDB storage for story prompt overrides.
- Key overrides by `storyId + promptId`.
- Compose global prompt + story instruction + locked contract + payload.
- Add reset story override to global default.

### Phase 4 — Prompt Manager upgrade

- Show prompt ID/version/category/scope.
- Show global vs story-specific mode.
- Show where each prompt is used.
- Add final rendered prompt preview.
- Add reset-to-default clearly.

### Phase 5 — Prompt test harness

- Allow testing a prompt with sample payload.
- Show parsed output and validation result.
- No save to canon unless user runs real workflow.

### Phase 6 — Expand to Persona and Role-play

- Add prompt groups only after Priority 1 is stable.
